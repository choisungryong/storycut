## 1. 현재 문제점 진단

### 치명적 문제
- **랜딩 페이지 부재**: 앱 접속 시 바로 폼이 보임 → 제품 가치 전달 불가
- **모바일 미대응**: `@media` 쿼리 없음, 심사위원이 모바일로 볼 가능성 높음
- **단일 HTML 1985줄**: index.html에 모든 섹션 + 인라인 스타일 → 유지보수 난이도 극상

### 디자인 품질 문제
- 기능 중심 UI → "개발자가 만든 느낌"
- CTA(Call-to-Action) 전략 부재
- 신뢰 요소 없음 (사용자 수, 후기, 데모 영상)
- 브랜딩 약함 (로고, 컬러 시스템, 타이포그래피)

### VMAI 대비 약점
| 항목 | VMAI | StoryCut |
|------|------|----------|
| 랜딩 페이지 | 전문 마케팅 페이지 (히어로/기능/후기/FAQ) | 없음 |
| 반응형 | 완벽 대응 | 미대응 |
| 시각적 세련도 | 그라디언트, 글래스모피즘, 마이크로 애니메이션 | 기본 다크 테마 |
| CTA 전략 | "무료로 시작하기" + 데모 영상 | 바로 폼 노출 |
| 신뢰 요소 | 사용자 수, 후기 카드, 사례 영상 | 없음 |
| 브랜딩 | 일관된 디자인 시스템 | 로고+버전 텍스트만 |

## 2. StoryCut 경쟁 우위 (VMAI에 없는 것)

이 기능들을 **시각적으로 강조**해야 함:

1. **뮤직비디오 생성** - VMAI에 없는 킬러 피처
2. **캐릭터 일관성** - Master Anchor + 7-step LOCK (업계 최고 수준)
3. **B-roll 스톡 영상** - Pexels 자동 배치 (비용 절감 + 품질 향상)
4. **7가지 아트 스타일** - anime, webtoon, hoyoverse, cinematic, realistic, illustration, abstract
5. **실시간 씬 편집** - 스토리 리뷰 → 이미지 프리뷰 → 개별 재생성
6. **AI 가사 타이밍 싱크** - 음원 분석 → 가사 자동 매칭
7. **I2V 영상 변환** - 이미지를 실제 영상으로 (Veo 3.1)

## 2-1. VMAI와의 차별화 전략

### 디자인 컨셉 차이
| | VMAI | StoryCut |
|---|------|----------|
| **포지셔닝** | "AI 비디오 자동화 서비스" (SaaS) | **"AI 영상 크리에이션 스튜디오"** (Creator Tool) |
| **비주얼 톤** | 클린 SaaS, 미니멀 | **시네마틱 스튜디오**, 필름스트립/타임라인 모티프 |
| **히어로** | 텍스트 + CTA + 데모 영상 | **듀얼 모드 분기** (스토리 영상 / 뮤직비디오) 인터랙티브 |
| **기능 소개** | 일반적 카드 그리드 | **라이브 파이프라인 시각화** (입력→스토리→이미지→영상 애니메이션) |
| **갤러리** | 없음 | **7가지 스타일 인터랙티브 갤러리** (탭 전환 + 실제 결과물) |
| **차별 피처** | 텍스트→영상 단일 | MV 생성, 캐릭터 일관성, B-roll, 가사 싱크 |

### 핵심 차별화 요소
1. **듀얼 모드 히어로**: 랜딩 히어로에서 "AI 스토리 영상" / "뮤직비디오" 두 갈래를 시각적으로 분기. VMAI는 단일 모드만 제공
2. **파이프라인 시각화**: "주제 입력 → AI 스토리 생성 → 이미지 생성 → 영상 합성" 과정을 스크롤 기반 애니메이션으로 시연. SaaS 랜딩에선 보기 드문 인터랙티브 요소
3. **스타일 갤러리**: 7가지 아트 스타일(anime, cinematic, webtoon 등)의 실제 결과물을 탭으로 전시. 크리에이터 도구다움을 강조
4. **필름스트립 UI**: 타임라인, 프레임, 컷 등 영상 제작 메타포를 UI 전반에 활용. "도구"가 아닌 "스튜디오" 느낌
5. **라이브 체험**: 랜딩에서 로그인 없이 간단한 데모 체험 가능 (1씬 무료 생성 등)

### VMAI에서 벤치마크할 것 (좋은 점은 배우되 다르게)
- 글래스모피즘 네비게이션 → 채용하되 시네마틱 톤으로
- 사용자 신뢰 요소 → "AI가 만든 실제 영상" 갤러리로 대체
- 모바일 반응형 → 필수 적용
- FAQ 아코디언 → 동일 패턴 채용

### 3-1. 페이지 분리

```

├── pricing.html          ← 기존 유지 + 디자인 통일
└── static/
    ├── landing.css       ← 신규: 랜딩 전용 스타일
    ├── landing.js        ← 신규: 랜딩 인터랙션
    ├── style.css         ← 기존 앱 스타일 개선
    ├── app.js            ← 기존 유지
    └── auth.js           ← 기존 유지
```

### 3-3. 앱 UI 개선 (app.html)

기존 워크플로우 유지하되 디자인 품질 향상:

#### 헤더 개선
- 로고 → SVG 아이콘 + 텍스트 조합
- 버전 번호 제거 (앱 내부에서만 표시)
- 네비게이션 정리: `새 영상` | `뮤직비디오` | `내 비디오` | `크레딧: 15`

#### 폼 UI 개선
- 2컬럼 레이아웃 (왼쪽: 설정, 오른쪽: 프리뷰/도움말)
- 스타일 선택: 텍스트 드롭다운 → 이미지 카드 선택
- 장르/분위기: 드롭다운 → 시각적 칩(chip) 선택
- "고급 옵션" 접기 → 깔끔한 기본 폼

#### 진행률 UI 개선
- 현재 프로그레스바 → 단계별 시각적 파이프라인
- 생성 중 씬 프리뷰가 하나씩 페이드인되는 갤러리 효과
- 예상 완료 시간 표시

#### 결과 화면 개선
- 비디오 플레이어 중앙 대형 배치
- 씬 타임라인을 플레이어 아래 가로 스크롤로
- 다운로드/공유 버튼 강조

## 6. 기술 스택 결정

### 옵션 A: 바닐라 HTML/CSS/JS (권장)
- **장점**: 현재 구조와 일관성, 추가 빌드 불필요, 배포 간단
- **단점**: 컴포넌트 재사용 수동
- **적합**: 해커톤 일정(~3주) 고려 시 가장 현실적

### 옵션 B: Tailwind CSS만 추가
- **장점**: 유틸리티 클래스로 빠른 스타일링, 반응형 내장
- **단점**: CDN 사용 시 빌드 불필요하지만 클래스가 길어짐
- **적합**: 디자인 속도가 중요한 경우

### 옵션 C: React/Next.js 전환
- **장점**: 컴포넌트 기반, 생태계
- **단점**: 전면 재작성 필요 (3주 내 불가능)
- **부적합**: 해커톤 일정에 맞지 않음

**결론: 옵션 A (바닐라) + Tailwind CDN 부분 활용**
